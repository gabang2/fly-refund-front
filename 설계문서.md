## 1. 현황 분석 및 개선 필요성

### 1.1 현재 엔진의 한계

현재 `src/lib/calculator/engine.ts`는 다음과 같은 심각한 단순화 문제를 가지고 있다.

| 항목 | 현재 구현 | 실제 필요 |
|------|-----------|-----------|
| 관할권 판단 | 항공사 이름 하드코딩 (4개 EU, 3개 US) | 출발지/도착지/항공사 국적 삼각 판단 |
| 거리 계산 | 도착 공항 IATA 코드 패턴 매칭으로 9000km/3000km 고정값 사용 | Haversine 공식 기반 실제 대원거리 계산 |
| 대한민국 규정 | 보상액 0원 (미구현) | 운임 기반 퍼센트 보상 (10%/20%/30%) |
| 지원 규정 | EU261, US DOT, 한국(미구현) 3종 | 최소 8개 관할권 규정 필요 |
| 취소 통보 타이밍 | 14일 이전/7일 전/당일 3단계 | 24시간/7일/14일/60일 다단계 분기 |
| 지연 처리 | 입력받지만 로직에 반영 안 됨 | 3시간 이상 지연 시 보상 트리거 |

### 1.2 비즈니스 기회

항공사보상정책.md에 따르면, 현재 미구현된 규정들이 다루는 시장은 다음과 같다:

- **캐나다 APPR**: 연간 수천만 명의 캐나다 출발/도착 여객
- **영국 UK261**: Brexit 이후 독립적 관할권, £220~£520 보상
- **인도 DGCA**: 급성장하는 인도 항공 시장
- **사우디 GACA**: 티켓 가치의 최대 200% 페널티 — 고액 보상 가능
- **베트남 Circular 14**: $25~$150 USD 고정 보상
- **브라질 ANAC**: 탑승거부 시 500 SDR (~€620)

## 2. 목표 아키텍처

### 2.1 전체 구조

```
사용자 입력
  ↓
[1단계] 관할권 라우팅 엔진
  ↓
[2단계] 거리 및 시간 계산 모듈
  ↓
[3단계] 항공사 분류 DB 조회
  ↓
[4단계] 귀책사유 분석
  ↓
[5단계] 보상액 산출 + 통화 변환
  ↓
결과 출력 (보상액, 적용 규정, 청구 가이드)
```

### 2.2 관할권 라우팅 로직 (Priority 순서)

```
IF dep_country ∈ EU_MEMBER_STATES OR dep_country ∈ {UK, IS, NO, CH}
  → EU261 또는 UK261 모듈

ELSE IF dep_country ∉ EU AND arr_country ∈ EU AND airline_country ∈ EU
  → EU261 모듈

ELSE IF dep_country = UK (non-EU departure) AND arr_country = UK AND airline_country ∈ {UK, EU}
  → UK261 모듈

ELSE IF dep_country = CA OR arr_country = CA
  → APPR 캐나다 모듈

ELSE IF dep_country = IN (인도 국내선)
  → DGCA 인도 모듈

ELSE IF dep_country = SA OR (arr_country = SA AND airline_country = SA)
  → GACA 사우디 모듈

ELSE IF dep_country = VN
  → Circular 14 베트남 모듈

ELSE IF dep_country = BR OR arr_country = BR
  → ANAC 브라질 모듈

ELSE IF dep_country = US OR arr_country = US
  → US DOT 환불 모듈

ELSE IF dep_country = KR OR arr_country = KR
  → 한국 소비자분쟁해결기준 모듈

ELSE
  → 몬트리올 협약 실비청구 안내
```

**관할권 중첩 처리:** UK ↔ EU 노선에서 EU 국적 항공사 운항 시, UK261과 EU261을 병렬 계산 후 실시간 환율 기준 더 유리한 결과를 우선 표시.

## 3. 핵심 모듈별 상세 설계

### 3.1 거리 계산 모듈

**현재 문제:** 도착지 IATA 코드를 정규식으로 매칭하여 9000km 또는 3000km로 고정.

**개선안:** Haversine 공식 기반 실제 대원거리 계산.

```typescript
// src/lib/calculator/distance.ts

interface AirportCoords {
  lat: number;
  lon: number;
}

// IATA 코드 → 좌표 DB (Supabase airports 테이블에서 조회)
export async function getGreatCircleDistance(
  depIata: string,
  arrIata: string
): Promise<number> {
  const [dep, arr] = await Promise.all([
    fetchAirportCoords(depIata),
    fetchAirportCoords(arrIata),
  ]);
  return haversine(dep, arr); // km 반환
}

function haversine(a: AirportCoords, b: AirportCoords): number {
  const R = 6371;
  const dLat = toRad(b.lat - a.lat);
  const dLon = toRad(b.lon - a.lon);
  const h =
    Math.sin(dLat / 2) ** 2 +
    Math.cos(toRad(a.lat)) * Math.cos(toRad(b.lat)) * Math.sin(dLon / 2) ** 2;
  return 2 * R * Math.asin(Math.sqrt(h));
}
```

**DB 스키마 추가 (Supabase migration):**

```sql
CREATE TABLE airports (
  iata_code   CHAR(3) PRIMARY KEY,
  name        TEXT NOT NULL,
  city        TEXT,
  country_code CHAR(2) NOT NULL,  -- ISO 3166-1 alpha-2
  lat         DECIMAL(9, 6) NOT NULL,
  lon         DECIMAL(9, 6) NOT NULL
);
```

> 초기 데이터: OpenFlights.org의 airports.dat 파일 (약 7,700개 공항 무료 제공)

### 3.2 EU261 / UK261 모듈

**파일:** `src/lib/calculator/modules/eu261.ts`

```typescript
interface EU261Input {
  distanceKm: number;
  delayHours: number;        // 도착지 문이 열린 시간 기준
  cancellationNotice: 'under7days' | '7to14days' | 'over14days' | null;
  reason: 'airline_fault' | 'weather' | 'extraordinary' | 'technical';
  reroutedArrivalDelay?: number; // 대체편 이용 시 지연 시간
  jurisdiction: 'EU261' | 'UK261';
}

interface CompensationResult {
  amount: number;
  currency: 'EUR' | 'GBP';
  regulation: string;
  halved: boolean;           // 50% 감액 여부
  careRights: CareRights;    // 돌봄의 권리 (식사/숙박)
  reason: string;            // 산출 근거 설명
}
```

**보상 매트릭스:**

| 거리 | EU261 | UK261 | 50% 감액 기준 (대체편 지연) |
|------|-------|-------|--------------------------|
| ≤1,500km | €250 | £220 | 2시간 이내 |
| 1,500~3,500km | €400 | £350 | 3시간 이내 |
| >3,500km | €600 | £520 | 4시간 이내 |

**면책 사유 분기:**

- `weather` / `extraordinary` (ATC 파업, 보안 위협, 공항폐쇄): 보상액 = 0, 돌봄의 권리는 유지
- `technical`: 보상 가능 (ECJ 판례: 일상적 기술결함은 면책 불가)
- `cancellationNotice === 'over14days'`: 보상액 = 0

**파일:** `src/lib/calculator/modules/appr.ts`

**핵심 변수 2개:** 항공사 규모(대형/소형) × 도착지 기준 지연 시간

**대형 항공사 DB 예시 (하드코딩 또는 Supabase airlines 테이블 플래그):**

### 3.5 사우디아라비아 GACA 모듈

보상 구조가 독특하므로 별도 모듈로 분리.

- **지연 보상:** 3~6시간 → 50 SDR, 6시간 초과 → 150 SDR

| 취소 통보 시점 | 추가 페널티 |
|----------------|------------|
| 14일~60일 전 | 티켓 가치의 50% |
| 24시간~14일 전 | 티켓 가치의 75% |
| 24시간 이내 | 티켓 가치의 150% |

- **SDR → KRW 변환:** IMF 일일 환율 API 연동 필요 (1 SDR ≈ 1,800~1,900원 변동)

### 5.1 입력 폼 추가 필드

| 필드 | 이유 |
|------|------|

| 티켓 구매 금액 | 한국 규정, 사우디 GACA 취소 보상 계산용 |
| 대체편 제공 여부 | EU261 50% 감액 적용 여부 판단 |
| 탑승거부 여부 | 오버부킹으로 인한 탑승거부는 별도 매트릭스 적용 |

### 5.2 결과 화면 개선

현재 결과 카드에서 추가해야 할 요소:

1. **적용 법 조항 배지:** "EU261/2004 제7조 적용" 형태로 법적 근거 명시
2. **돌봄의 권리 섹션:** 보상액이 0이더라도 식사/숙박비 청구 가능 여부 안내
3. **청구 성공률 표시:** 귀책사유 기반 예상 성공률 (예: "기술결함 → 청구 성공 확률 약 85%")
4. **관할권 중첩 알림:** UK261/EU261 중첩 시 두 금액 모두 표시, 유리한 쪽 추천
5. **청구 기한 안내:** "이 항공편의 청구 기한은 [날짜]까지입니다" (EU 2년, 한국 3년 등)
6. **몬트리올 협약 안내:** 고정 보상 불가 시 실비 청구 프로세스 안내

### 5.3 커뮤니티 포스트 연계

결과 화면에서 유사 노선/항공사의 성공 청구 사례를 커뮤니티에서 검색하여 표시:

```sql
SELECT p.* FROM posts p
WHERE p.airline = $airline
  AND p.route LIKE $route_pattern
  AND p.success = true
ORDER BY p.created_at DESC
LIMIT 3;
```

## 9. 리스크 및 고려사항

1. **법 개정 대응:** EU261 개정안 (지연 기준 완화, €500 상한)이 통과되면 항공편 날짜 기반으로 구법/신법 분기 필요. 알고리즘 설계 단계부터 날짜 기반 법 버전 선택 로직 포함 권장.

2. **한국 규정의 법적 강제력 부재:** 결과 UI에서 반드시 "권고 기준이며 항공사가 거절할 수 있음" 명시.

3. **실제 항공권 가격 수집 불가:** 한국/사우디 규정 등 운임 기반 보상 계산 시 사용자에게 직접 입력받아야 함. 이는 UX 부담을 높이므로, 선택적 입력으로 처리하고 미입력 시 최대 보상액만 표시.
